## What This Is

verl (Volcano Engine Reinforcement Learning) is a distributed RL training library for LLMs. It implements a "hybrid-controller" architecture using Ray for distributed execution, supporting PPO, GRPO, and other RL algorithms over training backends (FSDP, Megatron-LM) and inference backends (vLLM, SGLang).

## Architecture

### Core Concepts

verl uses a **single-controller** model: a Ray driver process orchestrates multiple distributed **worker** processes. Each worker hosts a model component (actor, critic, reference policy, rollout engine, reward model).

The key abstraction is `verl/single_controller/` which handles worker placement and remote method dispatch via Ray.

### Directory Map

```
verl/
  trainer/           # Entry points and training loops
    main_ppo.py      # PPO/GRPO/REINFORCE++ main entry (Hydra-configured)
    main_eval.py     # Standalone evaluation
    main_generation_server.py  # Offline generation
    sft_trainer.py   # SFT multi-controller entry
    ppo/
      ray_trainer.py # RayPPOTrainer: orchestrates all workers
      core_algos.py  # PPO/GRPO loss, advantage estimation, KL divergence
    config/          # Hydra config YAML files
      ppo_trainer.yaml  # Base config (do not edit _generated_*.yaml)

  workers/           # Distributed worker implementations
    actor/           # Policy update workers
    critic/          # Value function workers
    rollout/
      vllm_rollout/  # vLLM-based generation
      sglang_rollout/ # SGLang-based generation (supports multi-turn)
    engine/
      fsdp/          # FSDP training engine
      megatron/      # Megatron-LM training engine
    reward_manager/  # Reward computation workers

  tools/             # Tool calling for multi-turn RL
    base_tool.py     # BaseTool ABC
    search_tool.py   # Search/retrieval tool (SearchR1-like)
    utils/
      search_r1_like_utils.py  # HTTP client for retrieval server

  experimental/
    agent_loop/      # Multi-turn agent loop orchestration
      tool_parser.py # Parses tool calls from model output (registry pattern)
    fully_async_policy/  # Async actor-rollout decoupling
    one_step_off_policy/ # Off-policy correction

  models/            # Model architecture patches for Megatron
  utils/             # Shared utilities (dataset, reward_score, logging, etc.)
```

### Configuration System

verl uses **Hydra** for configuration. Configs live in `verl/trainer/config/` and are composed hierarchically. Override any value on the command line with `key=value` syntax.

The `_generated_ppo_trainer.yaml` is auto-generated by `scripts/generate_trainer_config.sh` — never edit it directly. Run `pre-commit run --all-files autogen-trainer-cfg` if config structure changes.

Key config groups:
- `actor_rollout_ref.*` — actor model, rollout engine, reference policy
- `critic.*` — value model
- `algorithm.*` — advantage estimator, KL settings
- `data.*` — batch sizes, sequence lengths, dataset paths
- `trainer.*` — epochs, logging, checkpointing

### Multi-Turn Tool Calling

For search-augmented RL (SearchR1-like), the flow is:
1. `RayPPOTrainer` calls the SGLang rollout worker with `multi_turn.enable=True`
2. The SGLang rollout invokes the `agent_loop` which uses a `ToolParser` (registry in `verl/experimental/agent_loop/tool_parser.py`) to detect tool calls
3. Tool calls dispatch to `verl/tools/search_tool.py`, which POSTs to a running retrieval server
4. The retrieval server (`examples/sglang_multiturn/search_r1_like/local_dense_retriever/retrieval_server.py`) uses FAISS + a dense encoder (e.g., e5-base-v2) against a Wikipedia corpus

Tool configs live in `examples/sglang_multiturn/config/tool_config/`. The rollout config key `actor_rollout_ref.rollout.multi_turn.tool_config_path` points to the active tool config.

### Logging

Set `VERL_LOGGING_LEVEL` (default `WARN`) to control verbosity of search/retrieval workers. Propagated automatically to Ray workers via `verl/trainer/constants_ppo.py`.

Experiment tracking: wandb, tensorboard, swanlab, mlflow — configured via `trainer.logger`.
